GIT常用命令和参数用法

Posted on 2020-07-22

GIT原理理解

Git在本地人为划分了工作区、暂存区和本地仓库
工作区即电脑里的工作目录下除开里面经过git init后创建的版本库外的所有其他文件
暂存区(storage)则存在于工作目录中隐藏的~~版本库中的index,因此也称之为索引
本地仓库(reopository)即版本库,工作目录中的隐藏.git目录

git status

git status 输出格式

本地活动 branch
本地branch连接的远程仓库branch

  1. Changes to be committed # 指已经从工作区上传到暂存区但未提交到版本库的文件
    (use “git restore –staged <file>…” to unstage)
  2. Changes not staged for commit # 指工作区,暂存区都保存了同一个文件,但是工作区修改过了
    (use “git add <file>…” to update what will be committed)
    (use “git restore <file>…” to discard changes in working directory)
  3. Unracked files #指修改过但未上传到暂存区的文件
    (use “git add <file>…” to include in what will be committed)

git add fileName

将文件从工作区保存到暂存区,支持一整个文件夹上传

git init

初始化一个文件夹为本地仓库(在该目录下创建一个隐藏的.git版本库)

git commit

常用参数:

  1. -m ‘message’
  2. -a 将所有跟踪文件中的修改或删除文件都提交到版本库,即使是没有添加到暂存区的文件
  3. –amend 在代码没有被merge之前,追加提交文件,可以不增加一个新的commit-id将新修改的内容追加到前一次的commit-id中

git branch

创建新分支

用指定的commit ID 创建新分支

git checkout

切换到已有分支

创建一个新分支并切换过去

git pull

git clone

从指定仓库以HTTPS链接形式克隆文件

从指定仓库以SSH形式克隆文件e

git rebase

git merge

用于版本控制
情景:当 A 修改了 repo 的 AR 分支时, B 同时修改了 repo 的主干分支,这时候 A 直接 pull 到主干不但会失败,还有可能会冲掉 B 修改的内容,于是要先 rebase
推衍合并分支,成功后merge分支内容进主干

git log –oneline

git push

情景1: 当远程仓库存在本地仓库没有的文件时,push失败并报错

操作:先从远程仓库上把文件都pull下来 git pull origin master --allow-unrelated-histories , git rm -r --cached target
git commit -m "delete target/"
git push origin master修改文件并 add 和 commit, 重新 push

Rebasing

对于多人协同开发来说，本地修改 commit (提交)后，在 push (推送)代码前，Rebasing 一下是一个很好的做法，log 看起来比较整齐，有助于阅读和理解（由于 merge 带来了蜘蛛网般的 log，想看清代码走向非常费劲; log 中常见的 “Merge branch ‘v0.7d-rebuild’ of http://git.example.com/wbs.wph into v0.7d-rebuild” ，本是不必要的 merge，也自然消失了）。
请不要对已经 push 出去的东西 再次 rebase 了。

理解 Rebase

假定把当前分支作为新特性分支，在开发过程中，主分支（Rebase 分支）又发生变化了，我们就要基于 Rebase 分支最新末端重新应用一下当前分支的每个 commit。
同理，rebase 概念适用于有跟踪关系的远程和本地分支的整合。

情景2: 当提交本地仓库到远程仓库后发现有些内容不必提交,希望远程仓库删除文件时
